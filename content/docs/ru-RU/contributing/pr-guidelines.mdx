# Руководство по оформлению pull-реквестов

Для обеспечения плавного и эффективного процесса проверки, как участники, так и проверяющие должны придерживаться следующих правил при отправке и рассмотрении pull-реквестов (PR).

## Для участников

1. **Информативные заголовки и описания**
   - Используйте четкие и лаконичные заголовки, кратко отражающие суть изменений.
   - Предоставьте подробное описание того, что делает этот PR, почему изменения необходимы, и любой соответствующий контекст.
2. **Ссылка на проблемы**
   - В заголовке вашего PR добавьте `GH-ISSUE-NUMBER` чтобы связать PR с соответствующей проблемой. Это требуется для всех PR.
3. **Небольшие, целенаправленные изменения**
   - Старайтесь делать PR небольшими и сосредоточенными на одной задаче или функции. Это упрощает понимание изменений и предоставление обратной связи проверяющими.
4. **Качество кода**
   - Убедитесь, что ваш код соответствует стандартам кодирования проекта и лучшим практикам.
   - Добавляйте комментарии там, где это необходимо для объяснения сложной логики.
5. **Тестирование**
   - Тщательно протестируйте свои изменения перед отправкой PR.
   - Если это применимо, включите модульные тесты или интеграционные тесты для проверки ваших изменений.
6. **Документация**
   - Обновите или добавьте документацию по мере необходимости, чтобы отразить внесённые изменения.
7. **Сообщения об изменения**
   - Использовать четкие и описательные сообщения об изменениях, которые следуют формату [conventional commits](https://www.conventionalcommits.org/en/v1.0.0/).
8. **Проверьте наличие конфликтов**
   - Перед отправкой PR убедитесь, что ваша ветка актуальна по отношению к базовой ветке, чтобы минимизировать конфликты слияния.

## Для проверяющих

- Вы должны внимательно изучить PR, и при необходимости запросить изменения.
- Предоставляйте конструктивную обратную связь и предложения по улучшению.
- Убедитесь, что изменения соответствуют стандартам кодирования проекта и лучшим практикам.
- Разверните PR с помощью интеграции dokploy, чтобы убедиться, что изменения работают корректно.
- Убедитесь, что документация обновлена по мере необходимости.
- Одобряйте PR только тогда, когда вы удовлетворены изменениями и все проверки пройдены.
